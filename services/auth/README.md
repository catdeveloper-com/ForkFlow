# Сервис аутентификации ForkFlow

## Текущий статус

Сервис можно запустить локально; он предоставляет только операционный маршрут
`GET /health`. Этот маршрут не содержит бизнес-логики и не подключается к
PostgreSQL или Redis.

В доменном слое уже описаны базовые строительные блоки регистрации:
email-адрес как value object и пользователь как доменная сущность. HTTP API
регистрации, хранение данных, хеширование пароля и проверка уникальности email
пока не реализованы.

В слое сценариев приложения добавлен use case регистрации пользователя. Он
создаёт доменный email, проверяет уникальность через порт хранилища и создаёт
пользователя. Реальный адаптер PostgreSQL пока не добавлен: в тестах
используется fake-репозиторий.

```bash
make run-auth
curl http://127.0.0.1:8000/health
```

Успешный ответ:

```json
{
  "status": "ok",
  "service": "auth"
}
```

## Конфигурация

Сервис читает настройки через `pydantic-settings`. Переменные окружения имеют
префикс `AUTH_`, а локальные значения можно положить в `.env`.

Сейчас используются только настройки метаданных FastAPI:

```env
AUTH_SERVICE_TITLE=ForkFlow Auth Service
AUTH_SERVICE_VERSION=0.1.0
```

Если переменные не заданы, сервис использует такие же значения по умолчанию.
Настройки базы данных, Redis, JWT и секретов пока не добавлены, потому что
сервис ещё не использует эти зависимости.

## Планируемая ответственность

Сервис аутентификации будет отвечать за идентификацию и аутентификацию:
регистрацию пользователей, проверку учётных данных, жизненный цикл токенов и
сессий, а также роли авторизации. Другие сервисы не должны напрямую обращаться
к его базе данных.

## Внутренние границы

Сервис следует направлению зависимостей Clean Architecture, принятому в
репозитории:

```text
presentation -> application -> domain
infrastructure -> application/domain through ports
```

- `domain` содержит независимые от фреймворков сущности, value objects и
  доменные ошибки.
- `application` содержит use cases, DTO и порты.
- `infrastructure` будет реализовывать хранение данных и внешние интеграции.
- `presentation` будет содержать HTTP-схемы и тонкие обработчики маршрутов.

Сейчас в коде есть `presentation` для healthcheck, `domain` для базовой модели
пользователя и `application` для сценария регистрации. `infrastructure` будет
добавлен, когда появится реальное хранение данных.

## Отложенная работа

Сценарии регистрации и входа, конфигурация внешних зависимостей, хранение
данных, миграции, токены и сессии должны появляться отдельными
сфокусированными изменениями.
