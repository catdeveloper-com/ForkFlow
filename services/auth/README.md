# Сервис аутентификации ForkFlow

## Текущий статус

Сервис можно запустить локально; он предоставляет только операционный маршрут
`GET /health`. Этот маршрут не содержит бизнес-логики и не подключается к
PostgreSQL или Redis.

```bash
make run-auth
curl http://127.0.0.1:8000/health
```

Успешный ответ:

```json
{
  "status": "ok",
  "service": "auth"
}
```

## Планируемая ответственность

Сервис аутентификации будет отвечать за идентификацию и аутентификацию:
регистрацию пользователей, проверку учётных данных, жизненный цикл токенов и
сессий, а также роли авторизации. Другие сервисы не должны напрямую обращаться
к его базе данных.

## Внутренние границы

Сервис следует направлению зависимостей Clean Architecture, принятому в
репозитории:

```text
presentation -> application -> domain
infrastructure -> application/domain through ports
```

- `domain` будет содержать независимые от фреймворков сущности, value objects
  и доменные ошибки.
- `application` будет содержать use cases, DTO и порты.
- `infrastructure` будет реализовывать хранение данных и внешние интеграции.
- `presentation` будет содержать HTTP-схемы и тонкие обработчики маршрутов.

Сейчас в коде есть только `presentation`: healthcheck не является доменным
сценарием и поэтому не требует сущностей, use case или репозитория. Остальные
слои будут добавлены, когда для них появится реальная ответственность.

## Отложенная работа

Сценарии регистрации и входа, конфигурация внешних зависимостей, хранение
данных, миграции, токены и сессии должны появляться отдельными
сфокусированными изменениями.
